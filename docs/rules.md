# Правила и законы проекта · CDP Generation

> Эти правила обязательны к соблюдению при любой работе в проекте. Нарушение требует явного согласия пользователя.

## I. Законы интеграций (повторение из spec.md)

1. **Context7** — единственный источник правды по документации библиотек. Перед написанием кода с зависимостью — обращение в Context7 MCP.
2. **Supabase** — только через Supabase MCP. Никаких прямых SQL в обход MCP.
3. **GitHub** — версионирование, ветки, PR, релизы только через `gh` CLI и `git`.

## II. Кодовые правила

### Архитектурные

- **Pipeline > монолит.** Этапы генерации — независимые шаги с типизированными контрактами на входе и выходе. Один шаг — один файл в `src/agents/`.
- **Стейтлесс агенты.** Каждый шаг получает данные на вход и возвращает данные на выход. Никаких глобальных переменных для состояния.
- **Конфиг через `.env` + Pydantic Settings.** Никаких хардкодов путей, ключей, моделей.
- **Без преждевременных абстракций.** Если есть один use case — пишем один класс. Обобщаем только при появлении второго.

### Стиль

- **Типы обязательны** во всех функциях и методах (`def f(x: int) -> str:`).
- **Современный синтаксис типов** — `list[str]`, не `List[str]`; `int | None`, не `Optional[int]`.
- **Pydantic v2** — для всех контрактов данных и конфигов.
- **`pathlib.Path`** вместо `os.path` для путей.
- **f-строки** вместо `.format()` или `%`.
- **Логирование через `logging`**, не `print` (кроме CLI-выводов).

### Имена

- `snake_case` для функций, переменных, файлов
- `PascalCase` для классов и Pydantic-моделей
- `SCREAMING_SNAKE_CASE` для констант
- Имена должны рассказывать **что**, а не **как**: `extract_client_facts()` лучше `process_data()`

### Комментарии

- **Без комментариев, описывающих *что* делает код** — имена идентификаторов уже это рассказывают
- **Комментарий уместен только когда объясняет *почему***: нетривиальное решение, обход бага, скрытое ограничение, ссылка на спеку
- Запрещены `# TODO:` без даты и автора. Лучше — issue в GitHub
- Запрещены закомментированные блоки кода в коммитах. Удаляй — git помнит

### Обработка ошибок

- **Без `try/except Exception:`** без явной обработки. Если ловим — знаем, зачем
- **Свои исключения** — наследуют от базового `CDPGenerationError`
- **Ошибки во внешних API** — exponential backoff, не больше 3 ретраев
- **Никаких пустых `except: pass`** в коммитах

### Зависимости

- **Перед добавлением новой зависимости** — проверить, нельзя ли решить стандартной библиотекой
- **Версии в `requirements.txt`** — фиксированные минимумы (`>=`), не плавающие или жёсткие
- **Перед использованием библиотеки** — Context7 MCP

## III. Правила Git и GitHub

### Ветки

- `main` — стабильная, защищённая. Прямые коммиты запрещены
- `feature/<short-name>` — новая функциональность
- `fix/<short-name>` — багфиксы
- `docs/<short-name>` — только документация
- `refactor/<short-name>` — рефакторинг без изменения поведения

### Коммиты

- Сообщения на английском или русском, **повелительное наклонение**: «add scenario validator», не «added/adds»
- Краткое summary в первой строке (≤72 символа), пустая строка, развёрнутое описание (если нужно)
- **Один коммит — одно изменение.** Не смешивать рефакторинг с фичей
- **Перед коммитом** — `git diff` глазами, проверить отсутствие секретов, отладочного кода
- **Trailer для AI-сессий:**

```
Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
```

### Pull Requests

- Заголовок ≤ 70 символов
- Тело: что сделано, как тестировалось, что оставлено на потом
- Линкуется issue, если есть
- Self-review перед запросом ревью

### Запреты

- ❌ `git push --force` в `main`
- ❌ `git commit --no-verify` без явного указания пользователя
- ❌ Коммиты с `.env`, ключами, токенами
- ❌ `git rebase -i` (интерактивный rebase) — не работает в неинтерактивной среде

## IV. Безопасность

1. **`.env` в `.gitignore`** — всегда. Проверять перед каждым `git add .`
2. **API-ключи** — только в переменных окружения, никогда в коде
3. **Логи** — не записывать содержимое запросов с персональными данными клиентов
4. **Перед коммитом** — `grep -r "sk-ant-\|sk-proj-\|TELEGRAM_BOT_TOKEN" src/` (или аналог) для проверки случайных токенов
5. **Supabase RLS** — обязательны для всех таблиц с пользовательскими данными

## V. Бизнес-правила (см. spec.md, повторно)

1. Цены: CDP «от 10 500 ₽/мес», Big Data «от 5 000 ₽/мес», скоринг включён
2. WABA — только если в брифе явно
3. Конкуренты — не упоминать по своей инициативе в выходных документах
4. Запрещённые отрасли — пометка на ревью
5. Реализуемость — только реальные триггеры/действия Calltouch

## VI. Тестирование

- **Юнит-тесты** для парсеров, валидаторов, моделей
- **Smoke-тест** pipeline на эталонных кейсах (LW Group, Level Group)
- Запуск перед коммитом: `pytest`
- CI: GitHub Actions запускает тесты на push в feature-ветки

## VII. Работа с субагентами

- **Используй проактивно.** Не жди явного запроса от пользователя — переключайся в субагента, когда задача попадает в его область
- **Не дублируй работу.** Если data-scout уже распарсил сайт — не парси повторно в scenario-author, а используй структурированный результат
- **Передача через типизированные контракты.** Между агентами — Pydantic-модели, не свободный текст

## VIII. Документация

- **README.md** обновляется при изменении quick start
- **CLAUDE.md** — при изменении правил работы AI
- **docs/spec.md** — при изменении контрактов или правил интеграций
- **`.claude/agents/<name>.md`** — при изменении ответственности субагента

## IX. Когда сомневаешься

1. Проверь `CLAUDE.md`
2. Проверь `docs/spec.md`
3. Проверь `docs/rules.md` (этот файл)
4. Проверь `memory/` через `MEMORY.md`
5. Проверь Context7 MCP
6. Если всё равно непонятно — **спроси пользователя, не выдумывай**

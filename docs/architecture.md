# Архитектура · двухфазная

## Общая логика

Проект развивается **последовательно** через две фазы. В фазу 2 переходим только после завершения и валидации фазы 1 на реальных кейсах.

## Фаза 1 · локально (текущая)

```
┌────────────────────────────────────────────────────────┐
│            МАШИНА ПОЛЬЗОВАТЕЛЯ (Windows)               │
│                                                        │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Streamlit веб-интерфейс (localhost:8501)       │   │
│  │  Фиолет + морской cyan, лого Calltouch          │   │
│  └────────────────┬────────────────────────────────┘   │
│                   │                                    │
│  ┌────────────────▼────────────────────────────────┐   │
│  │  Pipeline (Python 3.12)                         │   │
│  │                                                 │   │
│  │  1. data-scout      ──► Gemini Flash API        │   │
│  │  2. doc-keeper      ──► RAG поверх              │   │
│  │                         ../cdp_calltouch_kb/    │   │
│  │  3. scenario-author ──► Claude Opus API         │   │
│  │  4. visual-designer ──► Claude Opus API         │   │
│  │  5. reviewer        ──► Claude Opus API         │   │

              │                    │

       Claude API           Gemini Flash API
       (платно)             (free tier)
```

**Характеристики:**
- Запуск: `streamlit run src/app.py`
- Один пользователь, одна машина, без авторизации
- Данные в SQLite в папке проекта
- API-ключи в `.env`
- PDF-вывод в `./output/`

**Цель фазы 1:**
- Рабочий MVP, которым пользователь реально пользуется
- Минимум 5 реальных КП, сгенерированных инструментом
- Качество не хуже ручной работы

```
│    Telegram     │  Login   │   Frontend (Streamlit Cloud  │
│  Login Widget   │─────────►│   / Vercel / Render)         │
└─────────────────┘          │                              │

                             │   (allow-list по env)        │
                             └──────────┬───────────────────┘
                                        │
                             ┌──────────▼───────────────────┐
                             │   API (FastAPI / Streamlit)  │
                             │   + Pipeline (тот же)        │
                             └──────────┬───────────────────┘
                                        │
                ┌───────────────────────┼─────────────────────┐
                ▼                       ▼                     ▼
       ┌────────────────┐     ┌────────────────┐     ┌────────────────┐
       │   Supabase     │     │  Claude API    │     │  Gemini Flash  │
       │   - clients    │     │   (платно)     │     │   (free tier)  │

       │   - kp_history │
       │   - users      │
       │   - templates  │
       └────────────────┘
                ▲
                │ Storage: PDF/HTML результаты
                │
       ┌────────┴────────┐
       │ Supabase Storage│
       └─────────────────┘

           Deploy: GitHub Actions push → автодеплой
           CI: pytest + lint в feature-ветках
```

**Изменения относительно фазы 1:**

| Слой | Фаза 1 | Фаза 2 |
|---|---|---|
| Хостинг | localhost | Streamlit Cloud / Vercel / Render |
| БД | SQLite в файле | Supabase (Postgres + RLS) |
| Auth | нет | Telegram Login Widget + allow-list |
| Файлы | локальная папка `output/` | Supabase Storage |
| Деплой | вручную | `git push main` → GitHub Actions → автодеплой |
| Секреты | `.env` | переменные окружения хостинга + Supabase Vault |

**Цель фазы 2:**
- Доступ из любого браузера через Telegram-вход
- История КП и клиентов с возможностью повторно сгенерировать / посмотреть
- Бэкапы и версии

## Что не меняется между фазами

- Pipeline агентов — тот же
- Логика генерации — та же
- База знаний `cdp_calltouch_kb/` — общий источник
- Формат HTML/PDF — тот же
- Промпты — те же

Это намеренно: фаза 2 = это та же система, просто хостится в облаке. Не переписываем, а **разворачиваем**.

## Принципы перехода фазы 1 → 2

1. Не начинать работу над фазой 2 до зелёной фазы 1 (≥5 реальных КП)
2. Все файлы кода уже фазы 1 пишутся **с расчётом на дальнейший перенос** — без хардкода путей, конфиги в `.env`, SQLite через ORM с возможностью замены на Postgres
3. Перед стартом фазы 2 — отдельный спринт «миграция»: схема Supabase, GitHub-репозиторий, выбор хостинга, настройка Telegram-бота
